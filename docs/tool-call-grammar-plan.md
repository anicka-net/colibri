# Constrained decoding for tool calls

## Motivation

Tool calls are currently generated as ordinary text and parsed after generation.
This works for well-formed output, and the parser contains a few conservative
repairs for known model mistakes, but post-processing cannot reliably recover an
ambiguous call.

A captured Q4 failure illustrates the problem:

```text
<tool_call>bash</arg_value>printf 'KIT_TOOL_OK'</arg_value></tool_call>
```

The model chose the correct tool and retained the command value, but corrupted the
opening argument tags. Quantization and non-bit-exact execution make failures like
this probabilistic. The long-term fix is to constrain decoding after the model has
committed to `<tool_call>`, preventing malformed syntax rather than repairing it.

This is distinct from [grammar-forced speculative drafts](grammar-draft.md). That
feature proposes deterministic grammar bytes as drafts and verifies them with the
target model, deliberately preserving unconstrained output. Tool-call enforcement
must instead mask illegal next tokens and therefore changes the sampling domain.

## Intended behavior

Normal reasoning and answer text remain unconstrained. When the generated byte
stream completes `<tool_call>`, the sampler activates a request-specific grammar
compiled from the tools supplied by the client. The grammar remains active through
the matching `</tool_call>`, then returns to normal decoding. It can activate again
for a subsequent call.

While active, it must:

- accept only tool names present in the request;
- accept only argument names declared by the selected tool;
- reject duplicate argument names;
- require every schema-required argument before allowing `</tool_call>`;
- constrain values according to supported JSON Schema types;
- preserve the existing native XML wire syntax;
- stop cleanly if the connection is cancelled or the output limit is reached.

The first implementation should support the schema subset already understood by
Colibri's tool parser: objects with string, number, integer, boolean, null, array,
object, and enum-valued properties. Unsupported or ambiguous schemas must disable
enforcement for that tool rather than silently narrow its valid input space.

## Decoder design

### 1. Compile a request-specific automaton

At request admission, normalize OpenAI, Anthropic, Ollama, and Responses tool
descriptions into the existing internal representation. Compile tool names,
property names, required sets, types, and enum literals into an immutable grammar
owned by the request.

The runtime state includes:

```text
outside call
  -> tool name
  -> argument key or close
  -> argument value
  -> argument key or close
  -> outside call
```

It also tracks the selected tool and a bitset of arguments already emitted. This
semantic state is necessary because static GBNF alone cannot conveniently enforce
required fields and uniqueness.

### 2. Match tokenizer tokens by bytes

Tags, names, and JSON values do not necessarily align with tokenizer boundaries.
For each candidate token, feed its complete decoded byte string through a copy of
the automaton. The token is legal only if every byte is accepted and the resulting
state remains viable. Cache allowed-token sets by automaton state where possible;
value states with free-form strings will need a cheaper class/prefix cache.

Special tokens require an explicit policy. EOS is illegal while required arguments
or closing tags are outstanding. Stop tokens must not bypass the grammar in the
middle of a call, except cancellation and hard resource limits, which return an
explicit incomplete-call finish reason and never expose a partial call as valid.

### 3. Apply the mask at sampling

Insert the constraint after logits are produced and before temperature/top-p/token
selection. Illegal token logits become negative infinity. Greedy decoding then
selects the highest-scoring legal token; sampled decoding renormalizes over legal
tokens.

If no token is legal, fail the tool call explicitly and record the grammar state,
recent bytes, and schema identity in debug logs. Do not fall back to unconstrained
sampling inside an active call, because that recreates malformed structured output.

### 4. Keep parsing as validation

The existing incremental parser remains the API boundary and validates the emitted
call before translating it to each protocol. Conservative repair remains useful for
old checkpoints, unconstrained modes, interrupted streams, and unusual tokenizer
behavior, but grammar-constrained calls should pass without repair. Add counters for
calls constrained, calls validated, dead-end states, incomplete calls, and repairs.

## Delivery stages

1. **Automaton and schema compiler:** implement independently of inference, with
   byte-by-byte unit tests and rejection tests for unsupported schemas.
2. **Tokenizer bridge:** enumerate legal tokens for a state and test tags/tool names
   split across every possible token boundary.
3. **Greedy sampler integration:** enable behind `COLI_TOOL_GRAMMAR=1`; initially
   support one call and scalar values.
4. **Full supported schema subset:** add escaped strings, nested arrays/objects,
   enums, multiple calls, required fields, and duplicate rejection.
5. **All serving protocols:** verify identical behavior through OpenAI Chat,
   Responses, Anthropic Messages, and Ollama APIs, including streaming events.
6. **Default-on rollout:** compare tool-call success and latency on Q4 against the
   unconstrained baseline, then enable by default with an explicit opt-out.

## Test matrix

Unit tests must cover:

- every prefix of every XML delimiter;
- overlapping tool and argument names;
- required, optional, duplicate, and unknown arguments;
- strings containing escapes, XML-like text, and multibyte UTF-8;
- integer/number/boolean/null values, arrays, objects, and enums;
- EOS, stop sequences, cancellation, and token limits inside each grammar state;
- unsupported schemas taking the documented non-enforcing path;
- zero legal-token states producing an error rather than unconstrained output.

Integration tests should use a fake logits/token source to force the highest logit
onto illegal tokens and prove they are masked. End-to-end tests should run the same
tool prompts repeatedly on the Q4 checkpoint, record raw generation, and verify that
all API clients receive a valid call with complete arguments. Include KIT and Claude
Code scenarios, since both exercise streamed calls and multi-step tool results.

## Acceptance criteria

- No malformed call is delivered as a valid API tool event.
- All supported schemas produce parser-valid calls without repair.
- Required arguments cannot be omitted and unknown/duplicate arguments cannot be
  generated.
- Ordinary prose bytes and logits are unchanged outside tool calls.
- Streaming begins normally before a call and resumes normally afterward.
- The hot-path overhead outside calls is negligible; active-call overhead is
  measured and documented separately for prefill and decode.
- Debug logs can distinguish model choice, grammar masking, grammar dead ends,
  parser validation, and repair.

## Open questions

- Whether to retain XML internally or move the model-facing grammar to JSON while
  translating at the API boundary. XML matches GLM's learned format and is the safer
  first implementation.
- Whether constrained spans can also feed the existing grammar-draft mechanism.
  This may recover some masking overhead, but correctness should be established
  before combining the paths.
- How much nested JSON Schema support is justified. A small, explicit subset with a
  clean unsupported-schema path is preferable to incomplete enforcement presented
  as full JSON Schema support.
